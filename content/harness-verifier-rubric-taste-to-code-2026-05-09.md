블로그에 번역·기획·카드뉴스 같은 걸 AI한테 맡기면, 처음엔 결과물이 엉망임. 그러면 프롬프트를 고치고, 컨텍스트를 추가하고, 훅을 박고… 이 과정을 몇 번 반복하면 어느 순간 "이 정도면 쓸 만하네" 하는 지점에 도달함.

이 글은 그 **"쓸 만하다"를 어떻게 판단했는지, 그 판단 기준을 어떻게 기계에게 넘겨서 자동화하느냐**에 대한 이야기임.

---

## 1. 왜 이 글이 필요한가

요즘 AI 코딩 에이전트, 카드뉴스 메이커, 블로그 자동화... 다 "verifier(검증자)"가 핵심임. 그런데 verifier를 설계하라고 하면 대부분 막연함. "잘 만들어진 결과물"이 뭔지 우리는 알지만, 그걸 글로 적으려는 순간 손이 멈춤.

이게 폴라니가 말한 묵시지(tacit knowledge) — *"we know more than we can tell."* 좋은 디자인, 좋은 글, 좋은 카드뉴스를 알아보는 능력은 있는데, 그 기준을 명시할 줄 모름.

근데 하네스를 만들려면 그걸 명시해야 함.

---

## 2. 큰 그림: 하네스 토폴로지 5종

모든 하네스는 다음 5개 토폴로지의 조합임:

- **직렬** — 순차 파이프라인
- **병렬** — fan-out (throughput)
- **델리게이션** — sub-agent 위계
- **라운드로빈** — 인프라 분산 (계정/키)
- **작성-검증 반복** — writer-verifier loop

이 중 진짜 품질을 만드는 건 **작성-검증 반복** 하나. 나머지는 "더 빠르게/더 많이"고 이건 "더 정확히"임.

하네스 엔지니어링 = 사실상 verifier 엔지니어링.

---

## 3. Taste에서 시작함

오픈클로 만든 피터가 "taste"라고 부르는 게 있음. 결과물을 보고 "이건 아니다" 하고 걸러내는 직관. 이건 교육학으로 치면 평가자의 암묵지임.

카드뉴스 메이커를 예로 들면, 초기 루브릭은 이랬음:

- 내용에 어울리는 gif, 사진 사용
- 필요한 내용으로 이해하기 쉬운가
- 디자인이 UX를 고려하는가

**이건 taste지 루브릭이 아님.** "어울리는"이 누구 기준인지, "이해하기 쉬운"게 얼마나 쉬워야 하는지가 안 정의되어 있음.

---

## 4. 교육학에서 빌려오자: 백워드 디자인 (UbD)

평가를 직업으로 하는 사람들이 있다. 교사다.

Wiggins & McTighe의 **Understanding by Design (Backward Design)**:

> "활동을 먼저 짜지 말고, 평가를 먼저 설계해라. 그 평가를 통과하면 목표가 달성된다는 게 보장되도록."

이게 spec-first 하네스 설계랑 정확히 같은 구조:

| UbD (교육) | 하네스 |
|---|---|
| Stage 1: 학습 목표 | Task 정의 = taste |
| Stage 2: 평가 증거 | Rubric = verifier |
| Stage 3: 학습 경험 | Workflow chain |

순서가 같은 건 우연이 아님. 둘 다 **"모호한 목표를 관찰 가능한 신호로 분해하는 문제"**라서 같은 구조가 나옴.

대부분 하네스가 거꾸로 함. writer 먼저 만들고 verifier 나중에 붙임. 그러면 verifier가 writer 변형판이 되어서 같은 실수를 못 잡음.

---

## 5. 형성/총괄 → Working/Final Verifier

| 교육 | 하네스 |
|---|---|
| 형성평가 (formative) | Working verifier — 매 단계 체크 |
| 총괄평가 (summative) | Final verifier — 최종 게이트 |

분리되어야 함. 합치면 둘 다 대충됨. 형성은 빠르고 싸게, 총괄은 깊고 정확하게.

- 교육에서 "루브릭이 약하면 평가가 무너진다" = 하네스에서 "verifier가 약하면 에이전트가 무한히 돈다"
- 교육에서 "루브릭을 계속 개정한다" = 하네스에서 "eval을 돌려 verifier를 보강한다"

---

## 6. 묵시지를 루브릭으로 끄집어내는 4단계

### Step 1. 목표를 동사로 적기

- "좋은 카드뉴스" ❌
- "독자가 [무엇]을 할 수 있어야 하는가?" ✅

행동으로 정의하면 평가 가능해짐.

### Step 2. 차원으로 분해 (블룸 분류 활용)

Recall (사실 정확성) / Understand (이해도) / Apply (맥락 적합성) / Analyze (구조) / Evaluate (UX 판단) / Create (독창성)

각 차원별로 sub-rubric 1~3개.

### Step 3. 측정 가능한 형태로

- "어울리는가" ❌ → "키워드-이미지 의미 연결 Y/N" ✅
- 정성 형용사 금지

루브릭이 되려면 **관찰 가능한 신호**가 있어야 함:

**이미지 적절성**

| 점수 | 기준 |
|------|------|
| 0점 | 이미지 없음 |
| 1점 | 이미지 있으나 내용과 무관 |
| 2점 | 내용과 연관되나 어색함 |
| 3점 | 내용을 시각적으로 보완 |

**내용 이해도**

| 점수 | 기준 |
|------|------|
| 0점 | 주장 없음 |
| 1점 | 주장 있으나 근거 부족 |
| 2점 | 근거 있으나 구조 산만 |
| 3점 | 논리 구조 명확, 핵심만 전달 |

**UX 품질**

| 점수 | 기준 |
|------|------|
| 0점 | 텍스트 벽 |
| 1점 | 구분 있으나 가독성 낮음 |
| 2점 | 슬라이드당 정보량 적절, 폰트 일관 |
| 3점 | 시선 유도 자연, CTA 명확 |

### Step 4. 분산 테스트

같은 결과물에 verifier 5번 돌려봄. 점수 분산이 크면 루브릭 모호 → 더 분해.

---

## 7. 사례: 카드뉴스 메이커 루브릭

**원본 (모호):**

1. 내용에 어울리는 GIF/사진
2. 이해하기 쉬운가
3. UX 고려 디자인

**분해 후:**

**1) 시각자료**
- 키워드-이미지 의미 연결 (Y/N)
- 톤 일치 (진지/유머 매칭)
- 화질 720p+

**2) 콘텐츠 명료성**
- 슬라이드 1장 = 1메시지
- 타겟 독자 [명시] 기준 전문용어 적정
- 슬라이드 흐름 논리적

**3) UX 디자인**
- 모바일 폰트 16pt+
- WCAG AA 색대비
- 정보 위계 명확

빠진 거: **"목적이 정보전달/감정전달/행동유도 중 무엇인가"** — 목적 따라 가중치가 달라짐. 이게 UbD가 말하는 "목표 먼저"임.

---

## 8. Rubric → Verifier 구현: 측정 방법 라벨링

**핵심 질문: 이 루브릭을 뭘로 측정할 것인가?**

### [정량] — 코드로 측정 (가장 신뢰도 높음)

- 슬라이드당 이미지 ≥ 1개
- 텍스트 글자수 ≤ N자
- 폰트 크기 ≥ 16pt (모바일) / 24pt (데스크톱)
- 색상 대비비 ≥ 4.5:1 (WCAG 기준)
- CLIP score (이미지-텍스트 의미 정렬)
- 이미지 해상도 ≥ 720p
- 중복 이미지 비율
- 슬라이드 비율 9:16 또는 1:1

### [페르소나-LLM] — LLM으로 측정 (페르소나 필수)

정량만으로는 "톤이 맞는가"를 잡을 수 없음. 여기서 **페르소나 정의**가 중요. "누구한테 쉬운지"를 박아야 함.

- "25~35세 직장인이 5초 안에 핵심을 파악할 수 있는가"
- "이미지가 메시지의 톤(유머/진지/경고)과 일치하는가" 5점 척도

페르소나를 박는 순간 self-similar bias가 약해짐. taste를 "어디·누구한테냐"로 ground 시키는 셈.

### [사람] — 수동 채점 (교차 검증용)

5건 정도 직접 채점하고, LLM 채점 결과와 상관계수를 비교. **0.7 미만이면 루브릭이 taste를 아직 못 잡은 것.**

---

## 9. LLM-as-Judge의 함정

여기서 가장 흔한 실수: **verifier를 전부 LLM으로 만드는 것.**

**문제**: 채점 LLM이 작성 LLM과 self-similarity bias가 있음. 같은 맹점이 박혀있어서 채점이 자꾸 PASS가 나옴. 결국 본인이 결과물 보고 "이게 왜 PASS임?" 하며 한숨 쉬는 루프가 됨.

**GAN 비유의 한계도 여기에 있음**: GAN은 generator·discriminator가 같이 학습해서 discriminator가 generator 약점을 따라잡음. 하네스는 inference만 하고 학습이 없음. 그래서 LLM judge는 generator 약점을 영원히 못 따라잡음.

**해결은 단순함**: verifier 안에 **비-LLM 신호 비중**을 늘리는 것.

- 코드: test exit code, type-check, lint, build 성공
- 번역: COMET/BLEU 수치, length ratio, 고유명사 보존률
- 블로그: link liveness, spell-check, image alt 존재
- UI: 스크린샷 diff, DOM snapshot, WCAG 대비비

정량 신호 1개라도 박혀있으면 LLM judge가 무한 PASS를 못 냄. 정량이 FAIL인데 PASS를 낼 수 없으니까.

**"하네스 IQ 올리는 진짜 레버는 LLM 안 쓰는 verifier 비중"**임.

---

## 10. 묵시지 잔여물 (Taste Residue)

정직해야 할 부분. 루브릭으로 **80%**는 끄집어낼 수 있는데 나머지 20%는 안 됨. 인간 평가자도 본인이 왜 좋다고 했는지 분해 못 함. 이게 **taste residue**.

그래서 final human gate는 영원히 필요함. 하네스가 자동화하는 건 80%, 마지막 20%는 사람. 이걸 인정 안 하고 100% 자동화 시도하면 결과물이 **평균치로 수렴**함 — 평균보다 좋아질 수 없음.

---

## 11. 하네스의 전체 그림

이 verifier 설계는 하네스 엔지니어링의 outer loop 중 하나임.

**Inner loop** (프롬프트 영역):
- 프롬프트 설계
- 컨텍스트 관리
- 훅/미들웨어
- 재시도 루프 (ralph)

**Outer loop** (시스템 영역):
- **도구 설계** — generic이 아니라 작업 맞춤형 primitive
- **메모리 아키텍처** — 단기/장기/시맨틱 분리
- **검증 토폴로지** — verifier 여러 개를 직렬/병렬/계층으로 연결
- **상태 기계** — plan→work→review 단계 분리
- **복구 시멘틱스** — 멱등성, 부분 진행 보존, 보상 액션
- **관찰 가능성** — 사후 분석, replay

---

## 요약: 암묵지 → 루브릭 전환 체크리스트

1. ✅ 목표를 동사로 적었는가
2. ✅ 3~6개 차원으로 분해했는가
3. ✅ 각 항목이 Y/N 또는 척도로 측정 가능한가
4. ✅ working/final verifier 분리했는가
5. ✅ [정량 / 페르소나-LLM / 사람] 라벨링했는가
6. ✅ 5건 교차 채점으로 상관계수 0.7+ 확인했는가
7. ✅ 분산 테스트로 모호성 잡았는가
8. ✅ 마지막 human gate 남겨뒀는가

**하네스 설계 = 교육과정 설계. verifier = 루브릭. taste를 코드로 만드는 게 하네스 엔지니어링의 본질.**
