블로그에 번역·기획·카드뉴스 같은 걸 AI한테 맡기면, 처음엔 결과물이 엉망임. 그러면 프롬프트를 고치고, 컨텍스트를 추가하고, 훅을 박고… 이 과정을 몇 번 반복하면 어느 순간 "이 정도면 쓸 만하네" 하는 지점에 도달함.

이 글은 그 **"쓸 만하다"를 어떻게 판단했는지, 그 판단 기준을 어떻게 기계에게 넘겨서 자동화하느냐**에 대한 이야기임.

---

## 1. Taste에서 시작함

오픈클로 만든 피터가 "taste"라고 부르는 게 있음. 결과물을 보고 "이건 아니다" 하고 걸러내는 직관. 이건 교육학으로 치면 평가자의 암묵지임.

예를 들어 카드뉴스 메이커를 만든다고 하면, 초기 루브릭은 이랬음:

- 내용에 어울리는 gif, 사진 사용
- 필요한 내용으로 이해하기 쉬운가
- 디자인이 UX를 고려하는가

**이건 taste지 루브릭이 아님.** "어울리는"이 누구 기준인지, "이해하기 쉬운"게 얼마나 쉬워야 하는지, "UX 고려"가 구체적으로 뭔지가 안 정의되어 있음.

---

## 2. Taste → Rubric 변환

루브릭이 되려면 **관찰 가능한 신호**가 있어야 함. 각 항목을 채점 가능한 수준까지 내려가야 함.

### 예시: 이미지 적절성

| 점수 | 기준 |
|------|------|
| 0점 | 이미지 없음 |
| 1점 | 이미지 있으나 내용과 무관 |
| 2점 | 내용과 연관되나 어색함 |
| 3점 | 내용을 시각적으로 보완 |

### 예시: 내용 이해도

| 점수 | 기준 |
|------|------|
| 0점 | 주장 없음 |
| 1점 | 주장 있으나 근거 부족 |
| 2점 | 근거 있으나 구조 산만 |
| 3점 | 논리 구조 명확, 핵심만 전달 |

### 예시: UX 품질

| 점수 | 기준 |
|------|------|
| 0점 | 텍스트 벽 |
| 1점 | 구분 있으나 가독성 낮음 |
| 2점 | 슬라이드당 정보량 적절, 폰트 일관 |
| 3점 | 시선 유도 자연, CTA 명확 |

이 정도로 쪼개야 LLM verifier한테 줄 수 있고, 자동 채점이 가능해짐.

---

## 3. Rubric → Verifier 구현

여기서 핵심 질문: **이 루브릭을 뭘로 측정할 것인가?**

3가지 측정 방법이 있음:

### 정량 신호 (코드로 측정)

가장 신뢰도가 높음. LLM이 판단하는 게 아니라 기계적으로 참거짓이 결정됨.

- 슬라이드당 이미지 ≥ 1개
- 텍스트 글자수 ≤ N자
- 폰트 크기 ≥ 24pt
- 색상 대비비 ≥ 4.5:1 (WCAG 기준)
- 이미지 해상도 ≥ 특정값
- 중복 이미지 비율
- 슬라이드 비율 9:16 또는 1:1

### 페르소나-LLM (LLM으로 측정)

정량 신호만으로는 잡을 수 없는 "톤이 맞는가" 같은 영역. 여기서 페르소나 정의가 중요함. "누구한테 쉬운지"를 박아야 함.

- "25~35세 직장인이 5초 안에 핵심을 파악할 수 있는가"
- "이미지가 메시지의 톤(유머/진지/경고)과 일치하는가" 5점 척도

### 사람 (수동 채점)

초기 루브릭 검증용. 5건 정도 직접 채점하고, LLM 채점 결과와 상관계수를 비교. 0.7 미만이면 루브릭이 taste를 아직 못 잡은 것.

---

## 4. 교육과정과 하네스는 같은 구조임

이 과정을 교육학에서는 **Backward Design (UbD)** 이라고 부름:

| 교육과정 (UbD) | 하네스 설계 |
|---|---|
| Stage 1: 학습 목표 | Task 정의 = taste |
| Stage 2: 평가 증거 | Rubric = verifier |
| Stage 3: 학습 경험 | Workflow chain |

순서가 같음. 우연이 아니라 둘 다 **"모호한 목표를 관찰 가능한 신호로 분해하는 문제"**라서 같은 구조가 나옴.

여기서 한 발 더 나가면:

- 교육에서 "루브릭이 약하면 평가가 무너진다" = 하네스에서 "verifier가 약하면 에이전트가 무한히 돈다"
- 교육에서 "루브릭을 계속 개정한다" = 하네스에서 "eval을 돌려 verifier를 보강한다"
- 교육에서 "학생이 루브릭을 보고 자기평가" = 에이전트의 self-reflection 단계

---

## 5. LLM-as-Judge의 함정

여기서 가장 흔한 실수: verifier를 전부 LLM으로 만드는 것.

**문제**: LLM이 자기 결과물을 평가하면 self-similarity bias 때문에 무한 PASS를 낼 확률이 높음. "측정이 목표가 되면 더 이상 좋은 측정이 아니다" (Goodhart's Law).

**해결**: verifier 안에 **비-LLM 신호 비중**을 늘려야 함:

- 코드: test exit code, type-check, lint, build 성공
- 번역: COMET/BLEU 수치, length ratio, 고유명사 보존률
- 블로그: link liveness, spell-check, image alt 존재
- UI: 스크린샷 diff, DOM snapshot, WCAG 대비비

"하네스 IQ 올리는 진짜 레버는 LLM 안 쓰는 verifier 비중"임.

---

## 6. 실전: Rubric을 만드는 3단계

### Step 1: 쪼개기

각 항목을 sub 3~5개로 분해. "이미지 적절성" 하나에도 CLIP score, 존재 여부, 해상도, 중복 비율 같은 하위 항목이 나옴.

### Step 2: 측정 방법 라벨링

각 sub에 `[정량]`, `[페르소나-LLM]`, `[사람]` 태그 달기. 정량 비중이 높을수록 verifier가 강해짐.

### Step 3: 교차 검증

본인이 결과물 5건 직접 0~5점 채점 → LLM 채점과 비교 → 상관계수 확인. 0.7 미만이면 항목 추가/수정.

---

## 7. 하네스의 전체 그림

이 verifier 설계는 하네스 엔지니어링의 outer loop 중 하나임.

**Inner loop** (프롬프트 영역):
- 프롬프트 설계
- 컨텍스트 관리
- 훅/미들웨어
- 재시도 루프 (ralph)

**Outer loop** (시스템 영역):
- **도구 설계** — generic이 아니라 작업 맞춤형 primitive
- **메모리 아키텍처** — 단기/장기/시맨틱 분리
- **검증 토폴로지** — verifier 여러 개를 직렬/병렬/계층으로 연결
- **상태 기계** — plan→work→review 단계 분리
- **복구 시맨틱스** — 멱등성, 부분 진행 보존, 보상 액션
- **관찰 가능성** — 사후 분석, replay

이 중 verifier 설계가 교육과정 설계와 본질적으로 같은 작업이라는 게 이 글의 핵심.

---

## 요약

1. **taste를 종이에 적어라** — 안 적으면 암묵지 그대로고, 기계에게 넘길 수 없음
2. **taste를 루브릭으로 변환해라** — 관찰 가능한 신호까지 내려가야 함
3. **루브릭에 정량 신호를 박아라** — LLM-only verifier는 무한 PASS 함정
4. **교차 검증해라** — 사람 채점과 LLM 채점의 상관계수를 확인
5. **루브릭은 살아있는 문서** — 100건 돌리면서 계속 업데이트

하네스 설계 = 교육과정 설계. verifier = 루브릭. taste를 코드로 만드는 게 하네스 엔지니어링의 본질.
