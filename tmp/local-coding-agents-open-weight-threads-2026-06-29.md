## 1단 — 문제 제기

Codex와 Claude Code가 너무 좋아졌습니다.

그런데 바로 그래서 새 질문이 중요해졌습니다.

“이걸 전부 내 컴퓨터 안에서 돌릴 수 없을까?”

Sebastian Raschka의 글은 이 질문에 꽤 현실적으로 답합니다.

핵심은 단순히 Ollama 설치법이 아닙니다.

로컬 모델 + 추론 서버 + 코딩 에이전트 하네스 + 권한 모델 + 평가 방법을 한 묶음으로 봐야 합니다.

모델은 엔진이고,
하네스는 운전석입니다.

Qwen3.6 같은 모델을 Qwen-Code에 붙일 수도 있고,
Codex CLI에 붙일 수도 있고,
Claude Code에 우회 연결할 수도 있습니다.

그래서 로컬 코딩 에이전트의 질문은
“어떤 모델이 제일 똑똑한가?”가 아니라
“어떤 모델-하네스 조합이 내 작업에서 안전하고 효율적인가?”입니다.

---

## 2단 — 핵심 정리

로컬 코딩 에이전트가 매력적인 이유는 “무료”라서가 아닙니다.

진짜 장점은 통제권입니다.

- 민감한 코드와 문서를 외부 API로 보내지 않아도 됨
- 모델 업데이트로 워크플로우가 갑자기 깨지는 일을 줄임
- 오프라인/내부망 작업 가능
- 하네스의 파일 접근, 명령 실행, 텔레메트리를 직접 감사 가능

하지만 로컬이라고 자동으로 안전한 건 아닙니다.

코딩 에이전트는 파일을 읽고, 코드를 고치고, 셸 명령을 실행합니다.

그래서 처음 설치할 때는 최소한 이것부터 봐야 합니다.

- 설치 스크립트
- 셸 명령 실행 경로
- 파일 읽기/쓰기 범위
- 환경변수와 시크릿 상속
- MCP/플러그인
- 텔레메트리와 네트워크 호출
- 자동 업데이트

처음 쓰는 에이전트는 샌드박스 ON, 승인 모드 ON, 텔레메트리 OFF, 자동 업데이트 OFF가 기본값이어야 합니다.

## 3단 — 결론

지금 당장 Codex나 Claude Code를 버리고 로컬로 전부 갈아탈 필요는 없습니다.

오히려 현실적인 결론은 이쪽입니다.

주력은 클라우드 에이전트로 두되,
로컬 코딩 에이전트 스택을 “두 번째 작업대”로 만들어두는 것.

로컬 스택이 특히 유용한 순간은 분명합니다.

- 민감한 자료를 다룰 때
- API 한도나 비용이 걸릴 때
- 클라우드 모델 업데이트가 작업을 흔들 때
- 오프라인 환경에서 코딩해야 할 때
- 하네스 동작을 직접 감사하고 싶을 때

그리고 평가는 벤치마크 숫자만 보면 안 됩니다.

내가 자주 하는 작업 5~10개를 뽑아 직접 돌려봐야 합니다.

버그 재현, 테스트 추가, 리팩터링, 보안 리뷰, 실패 로그 분석 같은 실제 작업으로요.

로컬 코딩 에이전트의 의미는 “최고 성능”이 아닙니다.

내가 통제하는 두 번째 작업대.

클라우드가 막혀도,
요금제가 걸려도,
외부로 보내기 어려운 자료가 있어도,
계속 일할 수 있는 에이전트 환경입니다.
