들어가며: RL은 에이전트의 근육이다

2025년 초까지만 해도 “에이전트에 RL을 쓴다”는 말은 거의 실험실 이야기였다. SFT(Supervised Fine-Tuning)로 충분히 잘 돌아가니까. 하지만 2026년, 상황이 완전히 바뀌었다. 수학, 코딩, 도구 사용에서 RLVR(Reinforcement Learning with Verifiable Rewards)이 성능을 폭발시킨 뒤, 그 열기가 에이전트 영역으로 번지고 있다.

문제는 — 에이전트는 수학 문제처럼 정답이 깔끔하지 않다. 웹을 탐색하고, 터미널을 조작하고, 리서치 보고서를 쓰는 행동은 연속적이고, 보상은 희소하고, 환경은 비싸다.

이 글에서는 같은 주(5월 말)에 나온 세 편의 논문을 같은 비중으로 다룬다. 각자 에이전트 RL의 다른 측면을 공략하고 있다:

  1. CUA-Gym — 컴퓨터 사용 에이전트를 위한 훈련 환경을 대량 합성하는 파이프라인
  2. ECHO — 터미널 에이전트가 실패한 롤아웃에서도 학습하게 만드는 하이브리드 목적함수
  3. QUEST — 합성 데이터와 RL로 프론티어급 딥 리서치 에이전트를 훈련하는 전체 레시피

1. CUA-Gym: 환경이 없으면 에이전트도 없다

논문: CUA-Gym: Scaling Verifiable Training Environments and Tasks for Computer-Use Agents (Qwen 팀, 2026.05) 링크: arxiv.org/abs/2605.25624

왜 이 논문이 필요한가

컴퓨터 사용 에이전트(Computer-Use Agent, CUA)는 화면을 보고 클릭하고 타이핑하는 에이전트다. Claude Computer Use, Operator 같은 시스템이 대표적이다. 이 녀석들을 RL로 훈련하려면 (1) 과제 지시문, (2) 실행 가능한 환경, (3) 검증 가능한 보상 함수 세 가지가 동시에 필요하다.

기존에는 두 가지 극단만 있었다. 손으로 만든 벤치마크는 품질은 높은데 몇 개 안 되고, LLM-as-judge 방식은 스케일은 되는데 검증이 불안정하다. CUA-Gym은 이 사이의 빈 자리를 채운다.

어떻게 동작하는가

핵심은 세 에이전트의 협업이다.

Figure 1: CUA-Gym의 데이터 합성 파이프라인. Orchestrator가 Generator와 Discriminator를 조율하며, 정보 격리 벽을 통해 보상 해킹을 방지한다.

  1. Generator 에이전트 — 과제 지시문에 맞춰 초기 환경 상태(initial state)와 정답 상태(golden state)를 구성한다. “받은 편지함에서 제목에 ‘프로젝트’가 들어간 메일을 모두 읽음 처리해라” 같은 과제라면, 메일이 있는 상태와 읽음 처리된 상태를 각각 만든다.

  2. Discriminator 에이전트 — 같은 과제 명세를 받고 독립적으로 보상 함수를 작성한다. Generator가 만든 상태를 미리 알지 못하게 정보 격리(information isolation) 벽을 친다. 이게 핵심이다. 보상 함수를 짜는 에이전트가 정답 상태를 알면 보상 해킹(reward hacking)이 발생하기 쉽다.

  3. Orchestrator 에이전트 — 두 에이전트를 반복적으로 돌리며 실행 결과를 바탕으로 보완한다. 최종적으로 LLM 다수결 투표와 에이전트 롤아웃 필터를 거쳐 품질을 보증한다.

환경도 같이 만든다: CUA-Gym-Hub

훈련 환경이 부족하면 데이터가 아무리 많아도 소용없다. CUA-Gym-Hub는 실제 소프트웨어 사용 패턴을 기반으로 mock 웹 애플리케이션을 합성한다. Gmail 비슷한 메일 클라이언트, Slack 비슷한 메신저, Notion 비슷한 문서 편집기 등 110개 환경을 만들어냈다.

Figure 5: CUA-Gym으로 훈련한 모델의 OSWorld-Verified 벤치마크 결과. 도메인별 성능 향상을 보여준다.

결과

110개 환경, 32,112개의 검증된 훈련 튜플을 만들었다. GSPO로 훈련한 결과:

인터뷰로 정리하자면

Q: 환경 합성이 왜 이렇게 중요한가요?

“에이전트 RL의 병목은 모델이 아니라 환경이에요. 수학은 정답을 문자열로 비교하면 되지만, 컴퓨터 사용은 ‘실제로 클릭했는지, 파일이 생성됐는지’를 확인할 실행 환경이 필요하죠. 그걸 사람이 만들면 평생 걸립니다.”

Q: 정보 격리 벽이 핵심이라고 하셨는데?

“보상 함수를 짜는 에이전트가 정답을 알면, 보상 함수가 정답 상태를 하드코딩하는 식으로 타협해버려요. 그러면 에이전트는 과제를 해결하는 법을 배우지 않고 보상 함수를 해킹하는 법만 배우죠. Generator와 Discriminator를 분리하면 이 문제가 크게 줄어듭니다.”

2. ECHO: 실패한 롤아웃도 선생님이다

논문: ECHO: Terminal Agents Learn World Models for Free (Microsoft Research, 2026.05) 링크: arxiv.org/abs/2605.24517

문제의식: 85%의 롤아웃이 버려지고 있다

터미널 에이전트는 명령을 내리고, 터미널이 응답하고, 다시 명령을 내리는 식으로 동작한다. GRPO 같은 표준 RL에서는 action 토큰(에이전트가 생성한 명령어)에만 loss를 적용하고, environment 토큰(터미널의 응답)은 무시한다.

문제는 — Qwen3-8B 기준으로 85% 이상의 롤아웃이 과제를 해결하지 못한다는 것이다. 성공한 롤아웃만 학습에 기여하고, 실패한 85%는 버려진다. 하지만 실패한 롤아웃에도 파일 목록, 에러 로그, 빌드 실패 메시지, grep 결과 같은 풍부한 정보가 들어있다.

Figure 1: ECHO는 터미널 출력을 학습 신호로 활용한다. GRPO는 action 토큰에만 loss를 적용하지만, ECHO는 environment 관찰 토큰에도 cross-entropy loss를 추가한다.

핵심 아이디어: 환경 예측 = 세계 모델

ECHO(Environment Cross-entropy Hybrid Objective)의 수식은 놀라울 정도로 간단하다:

LECHO​(θ)=LGRPO​(θ;A)+λLEnv​(θ;O′)

A는 에이전트 action 위치, O′은 환경 관찰 위치. GRPO loss에 환경 토큰에 대한 cross-entropy loss를 λ=0.05 비중으로 더하는 게 전부다.

추가 롤아웃도, 추가 forward pass도, teacher 모델도 필요 없다. 이미 계산된 로짓에서 관찰 위치만 마스킹해서 loss에 더하면 된다.

왜 이게 효과가 있을까? 터미널 출력은 컨테이너 상태의 “손실 있는 텍스트 투영”이다. 파일 시스템 전체, 프로세스 트리, 히든 스테이트는 안 보이지만, stdout, stderr, exit code, 파일 내용은 보인다. 이걸 잘 예측하려면 에이전트가 터미널의 동작 방식을 이해해야 한다. 즉, 환경 예측 = 암묵적 세계 모델 학습이다.

결과: GRPO 성능을 2배로

설정TerminalBench-2.0 pass@1
Qwen3-8B + GRPO2.70%
Qwen3-8B + ECHO5.17%
Qwen3-14B + GRPO5.17%
Qwen3-14B + ECHO10.79%

거의 2배다. 그리고 더 흥미로운 건 — ECHO로 훈련한 모델은 자기가 생성하지 않은 롤아웃의 환경 토큰도 더 잘 예측한다. Qwen3-32B가 만든 held-out 롤아웃에서 ECHO 모델의 환경 cross-entropy가 급격히 떨어지지만, GRPO 단독은 거의 변하지 않는다.

전문가 데모 없이 SFT를 따라잡는다

Qwen3-8B 베이스에서 ECHO를 돌리면, 약 15,000개의 전문가 데모로 SFT한 뒤 GRPO를 돌린 것과 비슷한 성능이 나온다. 전문가 데모 없이 절반 정도의 SFT 이득을 회복하는 셈이다.

어떤 설정에서는 verifier(검증기) 없이도 환경 상호작용만으로 보이지 않는 OOD 과제에서 성능이 올라가는 현상도 관찰했다. 환경을 예측하는 것만으로 자기 개선이 가능하다는 뜻이다.

인터뷰로 정리하자면

Q: ECHO의 핵심 통찰이 뭔가요?

“터미널 응답이 ‘다음 행동을 위한 맥락’이 아니라 ‘학습 대상’이라는 거예요. ls를 쳤을 때 나오는 파일 목록은 입력이 아니라 정답지인 거죠. 에이전트가 ‘이 명령을 치면 저런 결과가 나오겠지’를 학습하면, 실패한 롤아웃에서도 엄청난 정보를 얻을 수 있어요.”

Q: λ=0.05면 너무 작은 거 아닌가요?

“환경 토큰이 액션 토큰보다 훨씬 많아요. 터미널 출력이 길수록 그 수가 압도적이죠. 0.05로 작게 해도 실제 그래디언트 기여는 상당합니다. 0.1만 넘어가도 정책 목표와 경쟁하면서 학습이 불안정해져요.”

3. QUEST: 8K 합성 과제로 프론티어를 넘는 법

논문: QUEST: Training Frontier Deep Research Agents with Fully Synthetic Tasks (OSU NLP, 2026.05) 링크: arxiv.org/abs/2605.24218

딥 리서치 에이전트란

딥 리서치 에이전트는 검색 엔진의 진화형이다. 키워드 매칭 페이지를 넘어서, 여러 소스를 탐색하고 정보를 종합하고 인용이 가능한 보고서를 작성하는 에이전트다. OpenAI의 Deep Research, Gemini의 리서치 기능이 상용 서비스로 나와 있지만, 오픈 소스 대안은 제한적이었다.

QUEST는 2B부터 35B까지의 모델 패밀리를 완전한 오픈 소스로 공개했다. 모델, 데이터, 훈련 스크립트 전부.

훈련 레시피: 3단계

QUEST의 훈련은 세 단계로 구성된다:

  1. Mid-Training (MT): 컨텍스트 요약과 관련 정보 추출 능력을 기른다. 약 130만 건의 데이터로, 긴 문서를 읽고 핵심을 뽑아내는 능력을 학습한다.

  2. Supervised Fine-Tuning (SFT): 두 종류의 과제를 학습한다.

    • 객관식(Objective) 과제: “2024년 미국 리스테리아 발생 건수는?”처럼 정답이 있는 팩트 체킹
    • 개방형(Open-ended) 과제: “애플의 폴더블 진출 전망”처럼 보고서를 작성해야 하는 과제

  3. Reinforcement Learning (RL): 합성된 과제로 RL 훈련. 여기서 Rubric Tree가 핵심이다.

Rubric Tree: 보상을 검증 가능하게

Figure 3: Rubric Tree 기반 데이터 합성 파이프라인. 객관식 과제는 Python 평가 스크립트로, 개방형 과제는 페어와이즈 비교로 검증한다.

RL의 고질적 문제: 에이전트가 잘했는지 어떻게 알까? LLM-as-judge는 불안정하고, 사람이 평가하면 스케일이 안 된다.

QUEST의 답은 Rubric Tree다. 과제를 평가 기준의 트리 구조로 분해한다:

  • 루트 노드: 과제의 핵심 질문
  • 중간 노드: 평가 차원 (정확성, 포괄성, 통찰력 등)
  • 리프 노드: 검증 가능한 개별 제약조건 (“출처를 인용했는가?”, “삼성과의 비교가 포함되었는가?” 등)

객관식 과제는 리프 노드 제약을 Python 스크립트로 자동 생성해서 평가한다. 개방형 과제는 페어와이즈 비교 방식으로 채점한다. 사람 주석이 전혀 필요 없다.

Context Condenser: 긴 호라이즌을 관리하는 법

Figure 4: Context Condenser가 긴 검색 세션의 컨텍스트를 압축한다. 검증된 사실은 유지하고, 모순된 주장은 낮추고, 불확실한 것은 추가 탐색을 유도한다.

딥 리서치는 수십 번의 검색, 수십 개의 URL 방문, 긴 추론을 거친다. 컨텍스트 윈도우를 금방 채운다. QUEST는 Context Condenser라는 메커니즘을 내장했다:

  • Trusted: 검증된 사실 (소스 URL 포함) → 직접 재사용
  • Untrusted: 모순된 주장 → 우선순위 하락
  • Uncertain: 추가 확인이 필요한 주장 → 새로운 검색 촉발

컨텍스트가 일정 한도를 넘으면 압축기가 작동하고, 에이전트는 깨끗한 컨텍스트로 작업을 이어간다.

결과: 8K 과제로 프론티어와 맞먹는다

모델BrowseCompGAIAWideSearchDRBLRB
GPT-559.976.454.0—73.1
Claude Opus 4.567.8—76.250.6—
Gemini 3 Pro59.2—57.049.6—
QUEST-35B45.580.860.648.268.2
QUEST-30B37.069.054.245.374.1

QUEST-35B는 GAIA에서 GPT-5, Claude, Gemini를 모두 제치고 80.8% 를 기록했다. QUEST-30B는 LiveResearchBench에서 GPT-5의 73.1%를 넘어 74.1% 를 달성했다. 오픈 소스 모델이 이 정도면 충격적이다.

그리고 RL 단계의 기여가 분명하다. 특히 개방형 리서치 과제(DRB, LRB)에서 RL 추가 후 성능이 크게 오른다. BrowseComp은 SFT만으로도 충분하지만, DeepResearch Bench는 SFT 36.4% → RL 48.2% 로 12%p 점프한다.

인터뷰로 정리하자면

Q: 8K 과제밖에 안 되는데 어떻게 이 성능이 나오죠?

“과제의 질이 중요하지 양이 중요한 게 아니에요. Rubric Tree가 보상을 정밀하게 검증하니까, 적은 과제로도 RL이 안정적으로 학습해요. 데이터 합성 파이프라인이 자동이니까, 필요하면 얼마든지 늘릴 수 있고요.”

Q: 오픈 소스 모델이 프론티어를 넘을 수 있다는 건가요?

“특정 벤치마크에서는 그렇습니다. GAIA나 LiveResearchBench에서는 이미 넘었어요. 하지만 BrowseComp 같은 팩트 검색에서는 아직 클로즈드 소스가 앞서요. 강점이 다르다고 보는 게 맞습니다.”

세 논문을 하나로: 에이전트 RL의 세 축

세 논문을 나란히 놓으면 에이전트 RL의 세 가지 핵심 축이 보인다.

축 1: 환경(ENVIRONMENT) — CUA-Gym

RL은 환경과의 상호작용에서 배운다. 환경이 없으면 RL도 없다. CUA-Gym은 이 가장 기본적인 전제를 대량 합성으로 해결했다. 과제, 상태, 보상 함수를 세 에이전트가 협업해서 만들고, mock 앱 환경까지 자동 생성한다. 환경의 스케일링이 에이전트 RL의 첫 번째 병목이었고, CUA-Gym이 이를 열었다.

축 2: 보상(REWARD) — ECHO

환경이 있어도 보상이 희소하면 학습이 안 된다. 터미널 에이전트의 85% 실패 롤아웃이 그 예다. ECHO는 롤아웃에 이미 있는 환경 관찰 토큰을 추가 학습 대상으로 삼아, 실패한 롤아웃에서도 밀도 있는 학습 신호를 얻는다. 추가 비용 없이, 환경 응답을 예측하는 능력(=세계 모델)을 부산물로 얻는다. 보상의 밀도화가 두 번째 축이다.

축 3: 데이터(DATA) — QUEST

환경도 있고 보상도 밀도 높은데, 훈련 과제가 부족하면? 특히 “딥 리서치” 같은 복잡한 과제는 사람이 만들기 어렵다. QUEST는 Rubric Tree 기반 합성 파이프라인으로 사람 주석 없이 훈련 데이터를 만든다. 그리고 합성 데이터 8K개만으로도 프론티어급 성능에 도달한다. 데이터 합성의 자동화가 세 번째 축이다.

정리: 에이전트 RL의 공식

에이전트 RL = 스케일러블 환경 × 밀도 있는 보상 × 자동 합성 데이터

CUA-Gym이 환경을, ECHO가 보상 밀도를, QUEST가 데이터 합성을 각각 해결했다. 세 축이 모두 “사람의 손을 빼는 방향”으로 움직이고 있다는 게 주목할 만하다. 환경 합성, 실패 롤아웃 활용, 자동 보상 생성 — 모두 사람 주석 없이 동작한다.

2026년 중반, 에이전트 RL은 더 이상 실험실 이야기가 아니다. 파이프라인이 갖춰지고, 성능이 증명되고, 모델과 데이터가 오픈 소스로 풀리고 있다. 다음 단계는 이 세 축을 하나의 시스템으로 통합하는 일이다. 환경 합성 파이프라인에 ECHO 같은 밀도 보상을 붙이고, QUEST의 Rubric Tree로 과제를 자동 생성하면 — 자가 진화하는 에이전트가 나올 수 있다.
