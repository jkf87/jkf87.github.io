OpenAI가 Codex 오케스트레이션을 위한 오픈소스 사양 Symphony를 공개했습니다. 핵심은 단순합니다. 사람이 여러 Codex 세션을 직접 붙잡고 지시하는 대신, Linear 같은 이슈 트래커를 코딩 에이전트의 컨트롤 플레인으로 쓰자는 것입니다.

공식 소개 글: https://openai.com/ko-KR/index/open-source-codex-orchestration-symphony/
GitHub 저장소: https://github.com/openai/symphony
SPEC 문서: https://github.com/openai/symphony/blob/main/SPEC.md

1. 왜 Symphony가 나왔나

OpenAI 팀은 이전에 “harness engineering” 방식으로, 사람이 직접 작성한 코드 없이 Codex가 코드를 생성하는 리포지토리 실험을 했습니다. 자동 테스트, 가드레일, 에이전트 친화적인 리포지토리 구조를 갖추면 Codex가 실제 팀원처럼 일할 수 있다는 것을 확인한 셈입니다.

그 다음 병목은 코드 작성 능력이 아니라 사람의 주의력이었습니다.

엔지니어가 Codex 세션을 3~5개 정도 동시에 관리하는 것은 가능하지만, 그 이상부터는 어떤 세션이 무슨 일을 하는지 기억하고, 중간에 멈춘 작업을 다시 밀어주고, 리뷰하고, 충돌을 해결하는 데 주의력이 급격히 소모됩니다.

OpenAI는 이 상황을 이렇게 봤습니다.

“우리는 매우 유능한 주니어 엔지니어 팀을 만들어놓고, 시니어 엔지니어에게 그들을 하나하나 마이크로매니징하게 하고 있었다.”

Symphony는 이 병목을 해결하기 위해 세션 중심이 아니라 업무 중심으로 관점을 바꿉니다.

2. Symphony의 핵심 아이디어

Symphony는 에이전트 오케스트레이터입니다. 이슈 트래커에 열린 작업이 있으면, 각 이슈마다 독립 워크스페이스를 만들고, 그 안에서 Codex App Server 기반 에이전트 세션을 실행합니다.

사람은 더 이상 “터미널 몇 개를 열고 에이전트에게 하나씩 지시하는 사람”이 아닙니다. 대신 이슈를 만들고, 우선순위를 정하고, 결과물을 리뷰하는 관리자 역할에 가까워집니다.

흐름은 대략 이렇습니다.

• Linear 같은 이슈 트래커에서 작업 후보를 읽습니다.
• 작업마다 별도 워크스페이스를 만듭니다.
• Codex App Server를 해당 워크스페이스에서 실행합니다.
• 리포지토리의 WORKFLOW.md에 정의된 프롬프트와 정책을 에이전트에게 전달합니다.
• 에이전트는 구현, 테스트, PR 생성, 리뷰 반영, 증거 자료 첨부까지 진행합니다.
• 사람은 최종 리뷰와 승인에 집중합니다.

OpenAI는 일부 팀에서 Symphony 도입 후 첫 3주 동안 머지된 PR 수가 500% 증가했다고 설명합니다. 숫자 자체보다 중요한 것은 병목이 “코드 작성”에서 “에이전트 작업 관리”로 이동했다는 점입니다.

3. 이슈 트래커가 컨트롤 플레인이 된다

Symphony의 가장 중요한 설계 선택은 Linear 같은 프로젝트 관리 보드를 에이전트 실행의 출발점으로 삼는 것입니다.

예를 들어 사람이 “이 코드베이스를 분석하고 마이그레이션 계획을 작성하라”는 이슈를 만들면, 에이전트가 먼저 계획을 만들고, 그 계획에서 다시 하위 작업 트리를 만들 수 있습니다. 작업 사이에 의존성이 있으면 DAG처럼 처리합니다. 막히지 않은 작업부터 병렬로 실행하고, 선행 작업이 끝나면 다음 작업이 시작됩니다.

OpenAI 글에서는 React 업그레이드 작업이 Vite 마이그레이션에 의존하도록 표시된 사례를 들었습니다. Symphony는 Vite 마이그레이션이 끝난 뒤에야 React 업그레이드 작업을 시작했습니다.

이 방식의 장점은 명확합니다.

• 사람은 작업 단위를 이슈로 표현하면 됩니다.
• 에이전트는 독립 워크스페이스에서 실행됩니다.
• 작업 상태, 리뷰, PR, 증거 자료가 이슈 중심으로 정리됩니다.
• 여러 작업이 자연스럽게 병렬화됩니다.
• 실패한 실험도 비용이 낮기 때문에 더 많은 탐색이 가능해집니다.

4. Symphony는 제품이라기보다 SPEC이다

GitHub 저장소를 보면 Symphony의 핵심은 거대한 플랫폼이 아니라 SPEC.md입니다. OpenAI는 Symphony를 “완성된 제품”으로 유지보수하려는 것이 아니라, 각 팀이 자기 환경에 맞게 구현할 수 있는 참조 사양으로 공개했습니다.

SPEC.md는 언어에 독립적인 서비스 사양입니다. 주요 컴포넌트는 다음과 같습니다.

• Workflow Loader: 리포지토리의 WORKFLOW.md를 읽고 설정과 프롬프트를 파싱합니다.
• Config Layer: poll interval, workspace root, 동시 실행 수, Codex 설정 등을 타입이 있는 값으로 제공합니다.
• Issue Tracker Client: Linear에서 후보 이슈와 상태를 읽고 정규화합니다.
• Orchestrator: polling loop, dispatch, retry, stop, reconciliation을 담당합니다.
• Workspace Manager: 이슈별 독립 워크스페이스를 만들고 생명주기를 관리합니다.
• Agent Runner: 워크스페이스 안에서 Codex App Server 클라이언트를 실행합니다.
• Logging / Status Surface: 여러 에이전트 실행을 운영자가 볼 수 있게 합니다.

중요한 경계도 분명합니다. Symphony는 스케줄러이자 러너입니다. 티켓 상태 변경, 댓글 작성, PR 링크 첨부 같은 구체적인 업무 로직은 WORKFLOW.md와 에이전트 도구 쪽에 둡니다.

5. WORKFLOW.md가 팀의 운영 계약이 된다

Symphony에서 인상적인 부분은 WORKFLOW.md입니다. 이 파일은 단순한 프롬프트가 아니라, 팀이 에이전트에게 맡기는 업무 방식의 계약입니다.

예를 들어 다음 내용을 담을 수 있습니다.

• 어떤 이슈 상태를 active로 볼 것인가
• 완료/취소/중복 같은 terminal state는 무엇인가
• 워크스페이스를 어디에 만들 것인가
• 새 워크스페이스 생성 후 어떤 hook을 실행할 것인가
• Codex App Server를 어떤 명령으로 띄울 것인가
• 에이전트가 이슈를 어떻게 읽고, 구현하고, 리뷰 증거를 남겨야 하는가

간단한 예시는 이런 형태입니다.

【코드: yaml】
tracker:
  kind: linear
  project_slug: "..."
workspace:
  root: ~/code/workspaces
hooks:
  after_create: |
    git clone git@github.com:your-org/your-repo.git .
agent:
  max_concurrent_agents: 10
  max_turns: 20
codex:
  command: codex app-server
【/코드】

그리고 Markdown 본문에는 에이전트에게 전달할 작업 프롬프트를 둡니다.

【코드: text】
You are working on a Linear issue {{ issue.identifier }}.

Title: {{ issue.title }}
Body: {{ issue.description }}
【/코드】

이 접근은 꽤 중요합니다. 많은 팀의 개발 워크플로우는 실제로는 사람들의 암묵지로만 존재합니다. “이슈를 잡고, 브랜치를 만들고, PR을 만들고, CI를 확인하고, 리뷰어에게 증거를 남긴다” 같은 절차가 문서화되어 있지 않은 경우가 많습니다.

Symphony는 이 암묵지를 WORKFLOW.md로 끌어냅니다. 그러면 에이전트도 그 절차를 따를 수 있고, 팀은 워크플로우 자체를 버전 관리할 수 있습니다.

6. Codex App Server가 왜 중요한가

Symphony는 Codex App Server 모드를 활용합니다. Codex를 CLI나 tmux 세션으로만 다루는 대신, JSON-RPC API로 스레드를 시작하고 턴을 관리하고 이벤트를 받을 수 있습니다.

이 점이 오케스트레이션에서 중요합니다. 여러 작업을 장시간 돌리려면 다음이 필요하기 때문입니다.

• 세션 시작과 종료를 프로그램으로 제어
• 토큰 사용량과 이벤트 추적
• stall timeout, retry, cancellation 처리
• 워크스페이스 격리
• 필요한 도구만 동적으로 노출

OpenAI는 Linear 토큰을 하위 에이전트 컨테이너에 직접 노출하지 않기 위해 dynamic tool calls로 raw linear_graphql 함수를 제공하는 방식도 언급했습니다. 즉, 오케스트레이션은 단순히 에이전트를 많이 띄우는 문제가 아니라 권한과 도구 노출 경계를 설계하는 문제이기도 합니다.

7. 참조 구현은 Elixir다

현재 저장소에는 Elixir/OTP 기반 참조 구현이 들어 있습니다. 이유는 분명합니다. Elixir/Erlang 생태계는 장시간 실행되는 프로세스, supervision tree, 동시성 처리에 강합니다.

Elixir 구현은 다음 흐름을 따릅니다.

• Linear에서 후보 작업을 polling
• 이슈별 워크스페이스 생성
• 워크스페이스 안에서 Codex App Server 실행
• workflow prompt 전달
• 작업이 끝날 때까지 Codex를 계속 진행
• terminal state로 이동한 이슈는 에이전트를 멈추고 워크스페이스 정리

다만 OpenAI는 이 구현을 “evaluation only” 프로토타입으로 명시합니다. 운영 환경에서는 SPEC.md를 기반으로 각 팀이 신뢰, 권한, sandbox, approval 정책을 맞춰 hardened version을 만드는 것이 권장됩니다.

8. Symphony가 보여주는 변화

Symphony가 흥미로운 이유는 “또 하나의 에이전트 도구”라서가 아닙니다. 개발 조직의 단위가 바뀌고 있기 때문입니다.

지금까지 코딩 에이전트 사용은 대체로 이런 방식이었습니다.

• 사람이 CLI를 연다.
• 에이전트에게 작업을 준다.
• 중간중간 방향을 수정한다.
• 결과를 확인한다.
• 다시 다른 세션을 연다.

Symphony식 접근은 다릅니다.

• 사람이 이슈를 정의한다.
• 에이전트가 가져간다.
• 오케스트레이터가 워크스페이스와 실행을 관리한다.
• CI, PR, 리뷰 증거가 붙는다.
• 사람은 결과와 방향을 판단한다.

즉 “에이전트와 대화하는 능력”보다 “에이전트가 일할 수 있는 업무 시스템을 설계하는 능력”이 중요해집니다.

9. 현실적인 한계와 주의점

물론 모든 작업이 Symphony에 맞는 것은 아닙니다. OpenAI도 애매하고 강한 판단이 필요한 문제는 여전히 엔지니어가 직접 interactive Codex 세션으로 다루는 편이 좋다고 설명합니다.

또한 에이전트를 티켓 단위로 맡기면 중간중간 세밀하게 steering하기 어렵습니다. 엉뚱한 결과물이 나올 수 있고, 그 실패를 수동으로 고치기보다 가드레일, 테스트, 문서, skill을 개선해야 합니다.

그래서 Symphony를 쓰려면 선행 조건이 있습니다.

• 테스트가 좋아야 합니다.
• CI가 신뢰 가능해야 합니다.
• 리포지토리 문서가 좋아야 합니다.
• 에이전트가 실행해도 되는 명령과 안 되는 명령의 경계가 있어야 합니다.
• 이슈 상태와 workflow가 명확해야 합니다.
• 사람의 최종 리뷰 루프가 있어야 합니다.

OpenAI가 말하는 harness engineering이 먼저 필요하고, Symphony는 그 다음 단계입니다.

10. 코난쌤 코멘트

Symphony는 “에이전트가 코드를 잘 짠다” 다음에 오는 문제를 정면으로 다룹니다. 이제 병목은 모델의 타이핑 속도나 코드 생성량이 아니라, 여러 에이전트가 동시에 일할 때의 관리 비용입니다.

개인적으로는 Symphony의 핵심을 세 가지로 봅니다.

첫째, 이슈 트래커를 에이전트 런타임의 control plane으로 본 점.

둘째, WORKFLOW.md로 팀의 암묵적인 개발 절차를 명시적인 계약으로 만든 점.

셋째, 완제품보다 SPEC을 공개해 각 팀이 자기 환경에 맞게 구현하도록 한 점.

앞으로 많은 회사에서 “코딩 에이전트 도입”의 다음 질문은 이렇게 바뀔 가능성이 큽니다.

“우리 팀의 이슈, PR, CI, 리뷰, 배포 흐름을 에이전트가 안전하게 따라갈 수 있게 문서화하고 오케스트레이션했는가?”

Symphony는 그 질문에 대한 OpenAI식 첫 번째 공개 답안입니다.

11. 내가 보는 Symphony의 개념과 장점

제 생각에 Symphony의 핵심 개념은 “코딩 에이전트용 작업반장”입니다.

지금까지 코딩 에이전트 사용은 사람이 Codex나 Claude Code 세션을 열고, 작업을 던지고, 결과를 보고, 다시 지시하고, PR을 확인하는 방식에 가까웠습니다. 이 방식은 에이전트가 1~2개일 때는 괜찮지만, 여러 개가 동시에 돌기 시작하면 사람의 주의력이 병목이 됩니다.

Symphony는 이 구조를 뒤집습니다. 사람이 에이전트 세션을 직접 관리하는 것이 아니라, 이슈 트래커가 에이전트 작업을 관리하게 만듭니다. 사람은 세션 관리자가 아니라 작업 관리자에 가까워집니다.

제가 보는 장점은 네 가지입니다.

첫째, 세션 관리에서 업무 관리로 바뀝니다.

기존 방식은 “Codex 세션 여러 개를 켜놓고 각각이 무엇을 하는지 기억하는 일”이었습니다. Symphony식 접근은 “Linear 이슈 상태를 보면 전체 작업 흐름을 알 수 있는 일”로 바꿉니다. 이 차이는 작아 보이지만, 실제 팀 운영에서는 큽니다. 사람의 머릿속 context switching 비용을 크게 줄이기 때문입니다.

둘째, 에이전트를 병렬 인력처럼 쓸 수 있습니다.

작업마다 독립 워크스페이스를 만들고, 막히지 않은 이슈를 동시에 처리합니다. 마이그레이션, 테스트 보강, 리팩토링, 문서화, 작은 버그 수정처럼 잘게 쪼갤 수 있는 작업에서는 특히 강합니다.

셋째, 팀의 암묵지를 WORKFLOW.md로 문서화합니다.

개발팀에는 말로 설명하지 않아도 모두가 알고 있는 절차가 많습니다. 브랜치를 만들고, 테스트를 돌리고, PR을 만들고, CI를 확인하고, 리뷰어가 볼 증거를 남기는 흐름 같은 것들입니다. Symphony는 이런 절차를 WORKFLOW.md라는 명시적인 계약으로 바꿉니다.

이 지점이 가장 중요하다고 봅니다. Symphony는 단순한 에이전트 실행기가 아니라, 개발 조직의 운영 절차를 코드화하는 도구입니다.

넷째, 실험 비용이 낮아집니다.

예전에는 “이 아이디어 한번 구현시켜볼까?”라는 생각이 들어도 사람이 세션을 열고 관리해야 했습니다. Symphony에서는 이슈 하나를 만들면 됩니다. 결과가 별로면 버리고, 괜찮으면 PR로 가져오면 됩니다. 덕분에 탐색적 작업을 더 많이 던질 수 있습니다.

다만 Symphony가 만능은 아닙니다. 테스트가 약하고, CI가 불안정하고, 이슈가 모호하고, 권한 경계가 흐릿한 팀에서 무작정 쓰면 “여러 에이전트가 동시에 사고치는 시스템”이 될 수 있습니다.

그래서 제 결론은 이렇습니다.

Symphony는 코딩 에이전트를 더 똑똑하게 만드는 기술이라기보다, 똑똑한 에이전트들이 팀 안에서 일할 수 있게 만드는 운영 체계입니다.

진짜 가치는 모델 성능이 아니라 작업 배분, 격리, 관찰, 리뷰, 반복 개선에 있습니다. 앞으로 에이전트 도입의 핵심 질문은 “어떤 모델을 쓰느냐”에서 “우리 팀의 업무 흐름이 에이전트가 안전하게 따라갈 수 있을 만큼 명시적인가”로 이동할 가능성이 큽니다.
