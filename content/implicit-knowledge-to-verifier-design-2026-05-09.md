## 하네스 Verifier 설계법

AI 에이전트 하네스를 만들다 보면 반드시 막히는 지점이 있다.

> "이게 잘 된 건지 어떻게 판단하지?"

대부분의 팀은 여기서 사람이 직접 결과물을 보고 판단한다. 그게 바로 **taste** — 경험에서 축적된 암묵적 판단력이다.

문제는 taste가 scale이 안 된다는 것이다. 에이전트가 하루에 100개 카드뉴스를 만들면 사람이 100개를 다 볼 수 없다. 그래서 verifier가 필요하고, 그 verifier를 잘 만들려면 먼저 암묵지를 종이에 끄집어내야 한다.

---

## 왜 대부분의 verifier가 실패하나

흔한 실수: LLM에게 "이 결과물이 좋은지 판단해"라고 시키는 것.

이건 두 가지 이유로 실패한다.

**1. self-similarity bias**
LLM-as-judge는 결국 자기와 비슷한 글을 좋다고 판단한다. generator와 verifier가 같은 모델이면 ceiling이 같아서 진짜 문제를 못 잡는다.

**2. 기준이 없으면 항상 PASS**
"좋은 카드뉴스인가?"라고 물으면 LLM은 거의 항상 긍정적으로 답한다. 판단 기준(rubric)이 없는 judge는 무한 PASS 기계다.

---

## taste → rubric → verifier

Wiggins & McTighe의 Backward Design과 구조가 같다.

1. **목표(taste)**: "나는 이런 결과물을 원한다"
2. **평가기준(rubric)**: "좋은 결과물의 조건은 X, Y, Z다"
3. **실행(verifier)**: 조건을 자동으로 측정하는 시스템

순서가 중요하다. 실행부터 만들면 verifier의 기준이 애매해진다.

---

## 실전: 카드뉴스 Verifier 설계

카드뉴스 메이커를 예로 들면, 처음 taste를 물어보면 이런 답이 나온다.

> "내용에 어울리는 gif/사진 사용, 이해하기 쉬운 내용, UX를 고려한 디자인"

이게 1차 rubric이다. 그런데 이대로 verifier를 만들면 전부 LLM-as-judge 영역이고, LLM은 무조건 PASS를 낸다.

각 항목을 **측정 가능한 sub-기준**으로 분해해야 한다.

### 디자인/UX (자동화 난이도: 낮음)

| 기준 | 측정방법 |
|------|----------|
| 폰트 ≥ 24pt | 코드 측정 |
| 텍스트-배경 대비비 ≥ 4.5:1 | WCAG 자동 검사 |
| 텍스트가 가장자리 6% 안 침범 | 픽셀 측정 |
| 슬라이드 간 폰트/팔레트 일관성 | diff 비교 |
| 9:16 비율 정확 | 코드 측정 |

이 항목들은 LLM 없이 코드로 100% 검증 가능하다.

### 이해하기 쉬운가 (자동화 난이도: 중간)

| 기준 | 측정방법 |
|------|----------|
| 슬라이드당 글자수 ≤ N | 코드 측정 |
| intro/body/outro 구조 존재 | binary 체크 |
| 핵심 키워드 coverage | 텍스트 분석 |
| 페르소나 기준 5초 요약 가능 | LLM-as-judge (페르소나 명시) |

여기서 핵심: **"누구에게 이해하기 쉬운가"**를 rubric에 박아야 한다. 페르소나 없이는 LLM도 기준이 없다.

### gif/사진이 내용에 어울리는가 (자동화 난이도: 높음)

| 기준 | 측정방법 |
|------|----------|
| 슬라이드당 이미지 ≥ 1개 | 코드 측정 |
| 해상도 ≥ N | 코드 측정 |
| 이미지-텍스트 의미 정렬 | CLIP score 또는 LLM-as-judge |
| 톤 일치 (유머/진지/경고) | LLM-as-judge (톤 명시) |

## Goodhart의 함정

측정이 목표가 되면 더 이상 좋은 측정이 아니다.

rubric 만점인데 실제 결과물은 기계적인 카드뉴스인 경우가 생긴다. 이게 Goodhart's Law다.

그래서 rubric은 taste의 **1차 근사**일 뿐이다. 사람이 주기적으로 "rubric 만점인데 별로였던 케이스"를 모아서 rubric을 업데이트하는 **메타 루프**가 필요하다.

이게 RLHF에서 reward model을 주기적으로 다시 학습시키는 이유와 같다.

---

## 실전 단계

1. taste를 3~5줄로 적어본다
2. 각 줄을 sub-기준 3~5개로 분해한다
3. 각 sub에 측정방법 라벨링: `[코드 / LLM / 사람]`
4. 코드 측정 가능한 것부터 verifier에 박는다
5. 결과물 5건을 직접 0~5점 채점하고 LLM 채점과 상관계수를 확인한다
6. 상관계수 0.7 미만이면 rubric이 아직 taste를 못 잡은 것 → 항목 추가/수정

---

## 핵심 원칙

> **하네스의 IQ는 verifier의 IQ다. verifier의 IQ는 LLM 없는 신호의 비중이다.**

LLM-as-judge 비중이 높을수록 실패를 못 잡는다. 가능한 한 코드 측정 가능한 정량 신호를 먼저 박고, 그 다음에 페르소나가 명확한 LLM-as-judge를 사용한다.

그리고 verifier는 한 번 만들고 끝이 아니다. taste가 진화하면 rubric도 진화해야 한다.
