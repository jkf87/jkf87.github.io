# Nomon: '평가 기준을 먼저 정하면 AI 코딩 에이전트가 달라질까?'

> 이 글은 연구 설계 단계의 정리입니다. 실제 실험 결과는 아직 나오지 않았습니다.

## 문제의식

AI 코딩 에이전트가 느는데, 우리는 이걸 어떻게 평가하고 있을까?

SWE-bench 같은 벤치마크에서는 "패치가 이슈를 해결했나?" (pass/fail)만 본다. 맞다/틀리다. 하지만 이 binary 평가에는 몇 가지 문제가 있다:

- **평가 기준이 사후에 정해진다.** 결과를 보고 나서 "이건 환각이었네"라고 판단한다. 평가가 개발과 분리되어 있지 않다.
- **정성적 기준이 없다.** 테스트는 통과했는데 근본 원인을 고친 건지, 아니면 증상만 가린 건지 구분이 안 된다.
- **LLM이 자기 점수를 매긴다.** LLM-as-judge는 유용하지만, LLM-only 평가가 100%면 자기 평가나 다름없다.

## Nomon의 제안: "루브릭 먼저, 실행은 나중에"

Nomon(노몬 — 해시계의 바늘, 그림자가 생기기 전에 방향을 정하는 역할)은 하나의 원칙으로 시작한다:

**평가 루브릭이 먼저 정의되고 검증되어야, 실행이 시작될 수 있다.**

구체적으로 4단계로 작동한다:

### 1단계: 루브릭 정의 (`rubric.yaml`)

각 과제에 대해 평가 기준을 YAML로 명시한다. SWE-bench Verified 기준으로 7개 기준을 사용한다:

| 기준 | 유형 | 가중치 | 측정 방법 |
|---|---|---|---|
| 단위 테스트 통과 | 정량 | 35% | 셸 실행 (exit code 0) |
| 패치 충돌 없이 적용 | 정량 | 15% | `git apply --check` |
| 보안 위반 import 없음 | 정량 | 5% | grep 스크립트 |
| 패치 크기 예산 내 | 정량 | 5% | diff 라인 수 카운트 |
| 이슈 완전 해결 | 페르소나-LLM | 25% | LLM 심사관이 근본 원인 해결 여부 평가 |
| 코드 품질 | 페르소나-LLM | 10% | LLM 심사관이 코드 스타일/복잡도 평가 |
| 설명 일관성 | 페르소나-LLM | 5% | LLM 심사관이 작성자 설명 정확도 평가 |

핵심 제약: **LLM-only 기준이 전체 가중치의 70%를 넘을 수 없다.** 최소 30%는 반드시 정량적(셸 실행, 테스트 결과 등)이어야 한다.

### 2단계: 검증자 드라이런 (verify-check)

실제 과제를 실행하기 전에, 가짜 입력(dummy input)으로 평가 파이프라인이 정상 작동하는지 확인한다. PASS/FAIL 신호가 결정적으로 나오지 않으면 실행이 차단된다.

이 단계가 왜 중요하냐면 — 루브릭에 버그가 있으면 수백 태스크를 돌린 뒤에야 발견하게 된다. 컴퓨팅 비용 낭비를 막는 안전망이다.

### 3단계: 작성자-검증자 루프 (writer-verifier loop)

이제 실제 과제를 실행한다:

1. 작성자 에이전트가 패치를 생성
2. 각 루브릭 기준이 독립적으로 채점
3. 가중합이 통과 임계값(0.70)을 넘으면 완료
4. 못 넘기면: 피드백을 작성자에게 돌려주고 재시도 (최대 3회)
5. 재시도 소진 시: 해당 과제는 실패로 기록

이 루프가 Nomon의 핵심 메커니즘이다. 환각(그럴싸하지만 틀린 패치)이 검증 단계에서 걸러지고, 재시도를 통해 수정된다.

### 4단계: 보정 (calibrate)

무작위 샘플에 대해 인간 평가자와 LLM 평가자의 점수를 비교한다. 피어슨 상관계수가 0.75 미만이면 해당 기준의 wording을 수정해야 한다.

## 실험 설계: 3조건 비교

Nomon의 효과를 검증하기 위해 같은 모델(`claude-sonnet-4-6`, temperature 0.2)로 세 조건을 비교한다:

**조건 A: Nomon (루브릭 우선)**
- 루브릭 정의 → 검증자 드라이런 → 작성자-검증자 루프
- 재시도 최대 3회
- 모든 루브릭 점수가 과제별로 로깅

**조건 B: Ouroboros (스펙 우선)**
- 이슈 설명으로부터 seed.yaml 생성 (구조화된 인터뷰 방식)
- 단일 패스 실행 (루브릭 게이팅 없음)
- 사후에 Nomon의 루브릭 스코어러로 평가 (공정한 비교를 위해)

**조건 C: Baseline (직접 LLM 호출)**
- 이슈 설명을 그대로 프롬프트로 입력
- 하네스 오버헤드 없음
- 마찬가지로 사후 루브릭 평가

왜 이 세 조건인가? Nomon의 핵심 가설이 "루브릭 우선 평가가 결과 품질을 높인다"이기 때문에, 스펙 우선(Ouroboros)과 아무 구조 없음(Baseline)을 대조군으로 둔다.

## 벤치마크: 왜 SWE-bench Verified인가

SWE-bench Verified 500개 태스크를 1순위로 선택한 이유:

- **가장 널리 쓰이는 에이전트 코딩 벤치마크** (2024-2026)
- **인간 검증됨** — 라벨 노이즈가 적다
- **Binary pass/fail**이 Nomon의 루브릭 게이트와 자연스럽게 매핑
- **SOTA 비교 가능** — Claude Code ~79.6%, Gemini ~80.6% 등 기존 결과와 직접 비교
- **MIT 라이선스** — 결과 공개에 제약 없음

2순위로 BigCodeBench 200개 샘플을 사용한다 (다양한 과제 타입 확보용).

## 측정 항목

| 지표 | 의미 | 측정 방법 |
|---|---|---|
| pass@1 | 첫 시도에 과제 해결 | SWE-bench 테스트 실행 |
| rubric_score | 루브릭 기준 가중평균 점수 | Nomon 루브릭 스코어러 |
| hallucination_rate | 환각 패치 비율 | 10% 샘플 수동 어노테이션 |
| tokens_per_task | 태스크당 총 토큰 사용량 | API 로그 |
| cost_per_task | 태스크당 USD 비용 | 토큰 × 모델 단가 |
| retry_count | 작성자-검증자 루프 반복 횟수 | Nomon 실행 로그 |

통계 분석:
- pass@1 비교: 쌍체 McNemar 검정 (태스크별로 짝지어서)
- rubric_score 비교: 3조건 ANOVA
- 유의수준 α = 0.05

## 파일럿 vs 풀런

**파일럿 (Phase 1):** 50 태스크 × 3 조건 × 3 반복
- 난이도별 층화 (Easy 17, Medium 17, Hard 16)
- 예상 비용: $30-50
- 목표: 프로토콜 검증, 루브릭 튜닝, 풀런 비용 추정

**풀런 (Phase 2):** 500 태스크 × 3 조건
- 예상 비용: $500-1,000
- 파일럿 결과에 따라 go/no-go 결정

## 아직 해결 안 된 결정사항

1. 파일럿 50태스크 먼저? 아니면 바로 700태스크 풀런?
2. HuggingFace 업로드용 토큰 (jkf87/nomon-swebench-eval-2026)
3. claude-sonnet-4-6 모델 확정?
4. 실험 인프라 셋업 (Docker + 데이터셋 로컬 복제) 진행 여부

## 알려진 한계

- **오염(contamination):** SWE-bench 이슈와 정답이 공개되어 있어 모델이 학습했을 가능성
- **스캐폴드 의존성:** 점수가 모델 자체 능력이 아니라 하네스 구조에 영향을 받음
- **인간 어노테이션 병목:** hallucination_rate는 10% 샘플만 수동 검토
- **비용:** 풀런이 $1,000 초과 가능
